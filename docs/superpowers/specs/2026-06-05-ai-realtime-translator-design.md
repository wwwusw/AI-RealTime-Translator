## 1. 项目背景

本项目面向“AI 同声传译助手”比赛题目，目标是在 Windows 场景下提供一个可直接使用的桌面工具，帮助用户在观看英语演讲、技术分享、国际会议或网课时，以较低门槛获得实时中文字幕，并具备对前文识别或翻译结果进行自动修正的能力。

本次第一阶段以 `2026-06-07 23:59:59` 为交付节点，优先目标按以下顺序排序：

1. 功能完整度
2. 易用性
3. 技术难度

## 2. 第一阶段目标

第一阶段交付一个可运行的 Windows 桌面 MVP，采用 `Electron + React + Node.js` 技术栈，重点完成以下能力：

1. 导入本地音频或视频文件。
2. 本地抽取音频并按时间窗口切片。
3. 调用云端服务完成英文转写、中文翻译。
4. 以滚动字幕形式展示结果。
5. 当后续上下文到来时，对最近几条字幕自动回改，体现“纠错能力”。
6. 提供基本的开始、暂停、重置、设置操作。

## 3. 第一阶段非目标

为保证在截止时间前稳定交付，以下内容不纳入第一阶段：

1. 系统音频实时采集。
2. 悬浮字幕窗。
3. 托盘常驻与全局快捷键。
4. 多任务并发处理队列。
5. 完整的字幕导出格式矩阵。
6. 完整离线推理。

其中“系统音频实时采集”会在架构上预留接口，作为第二阶段演进方向。

## 4. 技术栈与总体方案

### 4.1 技术栈

- 桌面壳：Electron
- 前端界面：React
- 运行环境：Node.js
- 音频处理：`ffmpeg`
- 翻译与纠错：DeepSeek API
- API 兼容策略：采用 OpenAI 兼容配置格式

### 4.2 为什么不先做纯网页

纯网页方案适合比赛演示，但不够接近真实用户长期使用场景。桌面工具更适合承载后续能力，例如系统音频采集、悬浮字幕窗、托盘常驻和快捷键控制，因此第一阶段直接采用桌面工具形态更符合“功能完整度 + 易用性优先”的目标。

## 5. 核心职责分工

第一阶段采用“本地处理管线 + 云端 Provider”的方案，职责如下：

### 5.1 语音提取

由本地 `ffmpeg` 完成：

1. 从本地音频/视频文件中抽取音频。
2. 统一转为适合转写的标准格式，例如单声道、16kHz PCM/WAV。
3. 按固定窗口切片，并保留少量重叠。

### 5.2 语音转文本

由独立的 `ASR Provider` 完成：

1. 输入音频切片。
2. 输出英文转写文本。
3. 如果服务支持，则同时保留时间信息与分段信息。

第一阶段只接入一套默认服务以先跑通流程，但接口必须独立抽象，避免后续替换时重构主流程。

### 5.3 翻译

由 `LLM Provider` 完成：

1. 输入英文分段文本。
2. 输出中文翻译字幕。

第一阶段默认使用 DeepSeek，配置方式采用 OpenAI 兼容格式：

- `baseUrl`
- `apiKey`
- `model`

默认模型优先使用 DeepSeek 当前推荐的新模型名，而不是即将弃用的旧模型别名。

### 5.4 纠错

也由 `LLM Provider` 完成：

1. 当新的英文上下文到来时，将最近几条英文内容与现有中文结果一起送入模型。
2. 模型只回改最近的窗口字幕。
3. 已冻结的旧字幕不再修改。

这样可以同时满足“自动纠错”和“用户阅读稳定性”两个目标。

## 6. OpenAI 兼容 Provider 设计

为了方便切换 DeepSeek 与其他兼容模型，翻译与纠错层统一采用 OpenAI 兼容结构，不在业务层写死某个厂商。

### 6.1 配置项

- `baseUrl`
- `apiKey`
- `model`
- 可选 `headers`
- 可选超时配置

### 6.2 接口目标

1. 第一阶段先接通 DeepSeek。
2. 后续可替换为其他 OpenAI 兼容模型而不修改字幕状态机和界面层。
3. 配置在设置页中填写并持久化存储。

## 7. 系统架构

### 7.1 模块划分

#### Electron Main

负责：

1. 应用窗口生命周期。
2. 文件选择。
3. 配置持久化。
4. IPC 通信。
5. 任务启动、暂停、重置。

#### React Renderer

负责：

1. 主界面渲染。
2. 文件导入与控制按钮。
3. 字幕列表与状态展示。
4. 设置页。
5. 纠错高亮与处理状态反馈。

#### Pipeline Worker

负责：

1. 调用 `ffmpeg` 抽音频与切片。
2. 驱动转写、翻译、纠错的处理流程。
3. 管理处理中、可回改、已冻结等字幕状态。

#### Provider Layer

负责：

1. 抽象 `ASR Provider`。
2. 抽象 `Translation/Revision Provider`。
3. 屏蔽厂商差异与请求格式细节。

#### Subtitle Store

负责：

1. 管理字幕条目。
2. 维护英文原文、中文翻译、状态、版本号。
3. 支持最近几条字幕回改。

#### Future Input Adapter

负责后续扩展：

1. 文件输入适配器。
2. 系统音频输入适配器。
3. 麦克风输入适配器。

第一阶段只实现文件输入，但接口形态要与未来实时音频输入兼容。

## 8. 数据流设计

第一阶段主流程如下：

1. 用户导入本地音频或视频文件。
2. 本地 `ffmpeg` 抽取标准化音频。
3. 将音频切为连续小片段，建议每段 `5s`，重叠 `1s`。
4. 每个音频片段先送入 `ASR Provider` 获取英文文本。
5. 英文文本进入字幕状态机，形成“暂定英文片段”。
6. 将最近若干英文片段送入 `LLM Provider` 生成中文翻译。
7. 当前端已有若干字幕后，每当新片段到来，触发一次“最近窗口回改”。
8. 当字幕超出回改窗口后，状态变为冻结，不再更新。

## 9. 字幕纠错机制

### 9.1 方案

采用“滑动窗口回改”策略。

### 9.2 规则

1. 只允许回改最近 `3-5` 条字幕。
2. 默认窗口设为最近 `4` 条字幕。
3. 超出窗口的字幕视为稳定结果并冻结。
4. 被回改的字幕在前端短暂高亮提示。
5. 避免整页全量重翻，降低阅读抖动。

### 9.3 为什么这样做

如果完全不纠错，无法满足比赛题目要求。  
如果每次都全量重翻，用户会持续面对大面积字幕跳变，阅读体验很差。  
“最近几条允许回改，旧字幕冻结”是兼顾准确率与可读性的折中方案。

## 10. 界面设计

### 10.1 界面结构

#### 顶部工具区

- 选择文件
- 开始处理
- 暂停处理
- 重置任务
- 打开设置

#### 中间主区域左侧

- 当前文件信息
- 时长

#### 中间主区域右侧

- 字幕时间轴

#### 底部状态区

- 当前 Provider
- 当前模型名
- API 连通状态
- 最近一次纠错提示

### 10.2 用户操作流程

1. 用户首次打开应用，进入设置页填写 `baseUrl`、`apiKey`、`model`。
2. 回到主页后导入本地音频或视频文件。
3. 点击“开始处理”后进入管线处理。
4. 字幕列表逐步滚动显示中文结果。
5. 最近几条字幕在有更多上下文后可能被短暂回改并高亮。
6. 用户可暂停、继续或重置任务。
7. 处理完成后保留完整字幕结果供查看。

## 11. 错误处理

第一阶段至少覆盖以下错误：

1. 未配置 API Key。
2. `baseUrl` 不可用或请求失败。
3. 模型名错误。
4. `ffmpeg` 不可用。
5. 文件格式不支持。
6. 单个片段转写失败。
7. 单个片段翻译失败。

处理原则：

1. 明确提示错误原因。
2. 尽量允许任务局部失败而不是整任务崩溃。
3. 对片段失败提供重试或跳过策略。
4. 不把底层异常原样暴露给普通用户。
